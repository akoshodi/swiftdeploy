# SwiftDeploy

SwiftDeploy is a manifest-driven deployment tool for a single API service behind Nginx.
`manifest.yaml` is the source of truth. All runtime configs are generated by the CLI.

## Repository Layout

- `manifest.yaml`: Declarative deployment config (edit this file only)
- `swiftdeploy`: CLI entry point with required subcommands
- `app/`: HTTP API service with stable/canary behavior and chaos controls
- `templates/`: `nginx.conf` and `docker-compose.yml` templates
- `Dockerfile`: Service image definition (lightweight, non-root)

## Prerequisites

- Docker Engine with Docker Compose plugin
- Python 3.10+

Install dependencies:

```bash
python -m pip install -r requirements.txt
```

Build the required local image referenced by `manifest.yaml`:

```bash
docker build -t swift-deploy-1-node:latest .
```

The image runtime uses only Python standard library modules, so container builds do not depend on downloading Python packages from PyPI.

If your build still fails because Docker itself cannot reach package repositories, retry with host networking:

```bash
docker build --network=host -t swift-deploy-1-node:latest .
```

If it still fails, fix Docker DNS on the host (for example by adding DNS servers in `/etc/docker/daemon.json`) and restart Docker.

## Manifest

The base required fields are present:

```yaml
services:
	image: swift-deploy-1-node:latest
	port: 3000

nginx:
	image: nginx:latest
	port: 8080

network:
	name: swiftdeploy-net
	driver_type: bridge
```

Additional fields (`services.mode`, `services.app_version`, `nginx.proxy_timeout`, etc.) are used for generation and lifecycle control.

## CLI Usage

Run from project root:

```bash
python swiftdeploy <subcommand>
```

### 1) `init`

Parses `manifest.yaml`, then generates:

- `nginx.conf`
- `docker-compose.yml`

```bash
python swiftdeploy init
```

### 2) `validate`

Runs exactly five pre-flight checks (non-zero exit on any failure):

1. `manifest.yaml` exists and is valid YAML
2. Required fields are present and non-empty
3. Service image in manifest exists locally
4. Nginx host port is not already bound
5. Generated `nginx.conf` passes `nginx -t`

```bash
python swiftdeploy validate
```

### 3) `deploy`

Runs `init`, then `validate`, then starts the stack and waits up to 60s for health.

```bash
python swiftdeploy deploy
```

### 4) `promote [canary|stable]`

Updates `services.mode` in `manifest.yaml`, regenerates configs, restarts only the app container, then confirms mode via `/healthz`.

```bash
python swiftdeploy promote canary
python swiftdeploy promote stable
```

### 5) `teardown [--clean]`

Stops stack and removes containers, networks, and volumes.
Use `--clean` to also remove generated files.

```bash
python swiftdeploy teardown
python swiftdeploy teardown --clean
```

## API Endpoints

- `GET /`: welcome payload with mode, version, timestamp
- `GET /healthz`: liveness and uptime seconds
- `POST /chaos`: enabled only in canary mode

Chaos payloads:

```json
{ "mode": "slow", "duration": 3 }
{ "mode": "error", "rate": 0.5 }
{ "mode": "recover" }
```

In canary mode, all responses include header `X-Mode: canary`.

## Generated Config Guarantees

- Nginx listens on `nginx.port`
- Proxy timeouts derive from `nginx.proxy_timeout`
- JSON error bodies for `502/503/504` with service/contact metadata
- `X-Deployed-By: swiftdeploy` header included
- `X-Mode` forwarded from upstream
- Access log format:
	`$time_iso8601 | $status | ${request_time}s | $upstream_addr | $request`

## Container Hardening and Runtime

- Service runs as non-root user
- Linux capabilities dropped (`cap_drop: [ALL]`)
- `no-new-privileges` set
- Health check on `/healthz`
- Service port is not exposed directly to host
- Traffic routed only through Nginx
- Named volumes used for logs
