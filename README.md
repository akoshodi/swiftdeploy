# SwiftDeploy

SwiftDeploy is a manifest-driven deployment tool for a single API service behind Nginx.
`manifest.yaml` is the source of truth. Runtime configs are generated by the CLI.

## Repository Layout

- `manifest.yaml`: Declarative deployment config (manually editable source of truth)
- `swiftdeploy`: CLI entry point with lifecycle, observability, policy, and audit commands
- `app/`: API service with stable/canary behavior, chaos controls, and Prometheus metrics
- `templates/`: `nginx.conf` and `docker-compose.yml` templates
- `policies/`: OPA Rego policies by domain
- `Dockerfile`: Service image definition

## Prerequisites

- Docker Engine with Docker Compose plugin
- Python 3.10+

Install dependencies:

```bash
python -m pip install -r requirements.txt
```

Build the required local image referenced by `manifest.yaml`:

```bash
chmod +x scripts/docker-preflight.sh
make docker-build
```

This runs a preflight check first to confirm Docker containers can resolve and reach PyPI. If DNS/networking is broken, it fails early with guidance before running the image build.

## Manifest

Base required fields (kept as requested):

```yaml
services:
  image: swift-deploy-1-node:latest
  port: 3000

nginx:
  image: nginx:latest
  port: 8080

network:
  name: swiftdeploy-net
  driver_type: bridge
```

Stage 4B adds declarative OPA and policy thresholds in `manifest.yaml` so Rego files stay environment-agnostic.

## Stage 4A + 4B CLI Commands

Run from project root:

```bash
python swiftdeploy <subcommand>
```

### `init`

Generates:

- `nginx.conf`
- `docker-compose.yml`
- `policy-data.json` (OPA data document generated from `manifest.yaml`)

It also validates that `policies/` contains `.rego` files.

```bash
python swiftdeploy init
```

### `validate`

Runs five pre-flight checks:

1. `manifest.yaml` exists and is valid YAML
2. Required fields are present and non-empty
3. Service image exists locally
4. Nginx port is not already bound
5. Generated `nginx.conf` passes `nginx -t`

```bash
python swiftdeploy validate
```

### `deploy`

Flow:

1. `init`
2. `validate`
3. Start OPA sidecar
4. Pre-deploy policy check via OPA (`infrastructure` domain)
5. Deploy app + nginx + opa if policy allows

If policy denies, deploy is blocked with explicit reasoning.

```bash
python swiftdeploy deploy
```

### `promote [canary|stable]`

Flow:

1. Start/check OPA
2. Scrape `/metrics` over policy window (default 30s)
3. Compute error rate and P99 latency
4. Pre-promote policy check via OPA (`canary_safety` domain)
5. If allowed, update `services.mode` in `manifest.yaml`
6. Regenerate configs and restart only app container
7. Confirm mode via `/healthz`

```bash
python swiftdeploy promote canary
python swiftdeploy promote stable
```

### `status`

Live-refreshing dashboard:

- Scrapes `/metrics`
- Calculates req/s and P99 latency over rolling 30s
- Evaluates policy compliance (infrastructure + canary safety) independently via OPA
- Appends every scrape to `history.jsonl`

```bash
python swiftdeploy status
python swiftdeploy status --interval 2 --samples 20
```

### `audit`

Parses `history.jsonl` and generates `audit_report.md` (GitHub Flavored Markdown):

- Timeline table (mode changes and chaos events)
- Policy violations section

```bash
python swiftdeploy audit
```

### `teardown [--clean]`

Stops stack and removes containers, networks, and volumes.
`--clean` also removes generated config/data files.

```bash
python swiftdeploy teardown
python swiftdeploy teardown --clean
```

## API Endpoints

- `GET /` welcome message with mode/version/timestamp
- `GET /healthz` liveness + uptime
- `POST /chaos` (canary only)
- `GET /metrics` Prometheus text format

Supported chaos payloads:

```json
{ "mode": "slow", "duration": 3 }
{ "mode": "error", "rate": 0.5 }
{ "mode": "recover" }
```

Canary mode adds `X-Mode: canary` to responses.

## Metrics Exposed

- `http_requests_total{method,path,status_code}`
- `http_request_duration_seconds` histogram
- `app_uptime_seconds`
- `app_mode` (0 stable, 1 canary)
- `chaos_active` (0 none, 1 slow, 2 error)

## OPA Policy Design

### Domain Separation

Policies are isolated by domain and queried independently:

- `swiftdeploy.infrastructure.decision`
- `swiftdeploy.canary_safety.decision`

A change in one domain does not require modifying the other domain.

### Threshold Source

Thresholds are not hardcoded in Rego. They are read from OPA `data.config`, generated by `swiftdeploy init` from `manifest.yaml` into `policy-data.json`.

### Output Contract

OPA returns structured decision objects with reasoning and violations, never a bare boolean.

## OPA Isolation / No Leakage

OPA is mapped to loopback only:

- `127.0.0.1:${OPA_HOST_PORT}:8181`

Nginx exposes only `nginx.port`; OPA is not routed through Nginx ingress.

## Required Policy Rules Implemented

- Infrastructure Policy: deny deploy when
  - disk free < 10GB (configurable)
  - CPU load > 2.0 (configurable)
- Canary Safety Policy: deny promote when canary and
  - error rate > 1% over last window (configurable)
  - P99 latency > 500ms over last window (configurable)
